# ktor-koin-template

An opinionated Ktor template for async Kotlin web services. Encodes an API-first approach, domain-centric architecture inspired by practical elements of DDD and Onion Architecture, and typed error handling with `kotlin-result`. Intended as a starting point for new services rather than a tutorial project.

## Tech Stack

**Framework & Runtime**

- Ktor 3.x (Netty engine)
- Kotlin 2.x, JDK 25 (Temurin)
- Koin 4.x (dependency injection)

**API**

- Fabrikt — generates controller interfaces and request/response models from OpenAPI spec
- Kotlinx Serialization (JSON)
- Swagger UI served at `/swagger`
- RFC 9457 Problem Details for structured API error responses

**Database**

- PostgreSQL 18
- jOOQ — type-safe SQL with code generation from live schema
- R2DBC + R2DBC Pool (non-blocking driver)
- dbmate — SQL migration runner

**Cache / Sessions**

- Valkey 8.1 (Redis fork) via Lettuce
- Session-based OAuth2 authentication (Discord provider)

**Authorization**

- OpenFGA — relationship-based access control (ReBAC)

**Observability**

- Cohort — health checks (`/health`)
- Micrometer + Prometheus (`/metrics`)
- Logback + kotlin-logging

**Testing**

- Kotest (FunSpec)
- Testcontainers (PostgreSQL + Valkey + OpenFGA)

**Quality**

- Detekt (static analysis)
- Kover (code coverage)

## Architecture

API-first: controller interfaces and request/response contracts are generated by Fabrikt from `src/main/resources/openapi/todo.yaml` (bundled from `contracts/todo/` by Redocly). Generated code lives in `src/generated/` — do not edit directly.

The architecture is domain-centric, taking practical inspiration from DDD and Onion Architecture. All business logic in the service layer operates exclusively on domain models. Both the API and the database are treated as edges with their own type systems — generated contracts and jOOQ records respectively — and explicit mappers at each boundary translate between those edge types and the domain.

```
┌─────────────────────────────────────────────────────────┐
│  API edge                                               │
│  OpenAPI spec → Fabrikt → controller interface          │
│  Types: *Contract (generated)                           │
│  Mapper: *ContractMapper (contracts ↔ domain)           │
└───────────────────────┬─────────────────────────────────┘
                        │ domain models only
              ┌─────────▼──────────┐
              │  Domain            │
              │  *Service          │
              │  kotlin-result,    │
              │  transactions      │
              └─────────┬──────────┘
                        │ domain models only
┌───────────────────────▼─────────────────────────────────┐
│  Database edge                                          │
│  *Repository (jOOQ queries)                             │
│  Types: *Record (jOOQ-generated)                        │
│  Mapper: *Mapper (domain ↔ records)                     │
│  PostgreSQL via R2DBC                                   │
└─────────────────────────────────────────────────────────┘
```

Each feature is a self-contained module under its own package (e.g. `todo/`, `todolist/`), each with its own controller, service, repository, mappers, and validation. Shared utilities live in `core/` (pagination helpers, validation rules, repository error handling).

Each layer has a typed error type. Repository returns `RepositoryResult<T>`, service returns a module-specific result type (e.g. `TodoListServiceResult<T>`), and the controller converts service errors to `ProblemDetailsException` at the HTTP boundary, producing RFC 9457 Problem Details responses. The `kotlin-result` library (not stdlib `Result`) is used throughout.

`resultTransactionCoroutine` is a custom extension wrapping jOOQ's `transactionCoroutine` that automatically rolls back the transaction when the inner block returns `Result.Err`.

Koin modules are defined per feature (e.g. `todoModule`) and assembled in `config/Koin.kt`. Infrastructure modules (database, Redis, monitoring) are also in `config/`.

## Authorization

Access control is relationship-based (ReBAC) via [OpenFGA](https://openfga.dev/), not ownership columns in the database. Users are granted `owner`, `editor`, or `viewer` on a `todo_list`; those relations compute `can_read`/`can_write`/`can_delete`, which individual todos inherit from their parent list. The model itself is defined in [`fga/authorization-model.fga`](fga/authorization-model.fga) — that file is the source of truth, not this README.

Today, relation assignment happens only at the `todo_list` level — a `todo` has no relations of its own, so it can't be shared independently of its list. That's a deliberate simplification for this template rather than a limitation of the approach: direct per-todo relations (e.g. `owner`/`editor`/`viewer` on `type todo`, unioned with the inherited ones) could be added later purely as a model change, since the application code already checks permissions against the specific resource being acted on and writes relationship tuples generically.

`AuthorizationService` (backed by `OpenFgaAuthorizationService`) exposes `check`, `listResourceIds`, `writeTuples`, and `deleteTuples`. A failed permission check returns 404 if the user has no access at all, or 403 if they can view the resource but lack the specific permission for the attempted action. Ownership tuples are written and removed inside the same database transaction as the resource itself, so a failed OpenFGA write rolls back the whole operation rather than leaving an inaccessible, orphaned row.

List endpoints (`GET /todo-lists`, `GET /todos`) use `listResourceIds` to fetch everything the caller can read, not just what they created. This template's implementation calls OpenFGA's ListObjects API live at read time, which is simple but caps out around ~1000 objects per call and doesn't scale well once a user's accessible set grows large. A production service at that scale should instead maintain its own denormalized "who can see what" index — most simply a DB join table — kept up to date by consuming OpenFGA's tuple-change feed (`/changes`), and intersect that index with the normal DB query at read time rather than calling ListObjects per request.

The OpenFGA store and model are provisioned automatically — by the `fga-provision` service in `docker compose up`, and by equivalent Testcontainers in integration tests.

## Database Migrations

Managed by [dbmate](https://github.com/amacneil/dbmate). Migration files in `db/migrations/` using timestamped names. Schema dump at `db/schema.sql`, maintained automatically by dbmate.

```bash
dbmate up              # Apply pending migrations
dbmate new <name>      # Create a new migration file
./gradlew jooqCodegen  # Regenerate jOOQ classes from live DB schema (requires DB at localhost:5432)
```

## Testing

**Unit tests** (`src/test/`) — Kotest FunSpec, no infrastructure needed:

```bash
./gradlew test
```

**Integration tests** (`src/integrationTest/`) — full-stack with real DB and Redis:

```bash
./gradlew integrationTest
```

- Testcontainers spins up PostgreSQL 18 + Valkey 8.1 + OpenFGA automatically
- dbmate migrations are applied inside the containers before tests run
- Each test truncates tables in `beforeEach` for isolation
- Tests exercise the full HTTP stack via Ktor's `testApplication`
- Includes user-isolation tests verifying cross-user data access is denied

Docker must be running for integration tests.

## Running Locally

```bash
# Start dependencies (DB, Redis, OpenFGA, run migrations)
docker compose up db valkey dbmate openfga fga-provision

# Run the app
./gradlew run
```

Or run the full stack:

```bash
docker compose up
```

App starts on `http://localhost:8080`. Swagger UI at `http://localhost:8080/swagger`.

| Command | Description |
|---|---|
| `./gradlew build` | Compile, unit tests, integration tests, detekt |
| `./gradlew test` | Unit tests only |
| `./gradlew integrationTest` | Integration tests (requires Docker) |

| `./contracts/build.sh` | Bundle OpenAPI contracts → `src/main/resources/openapi/todo.yaml` (requires Docker) |

## Code Generation

Do not edit `src/generated/` — it is regenerated by:

- `./gradlew generateApi` — Fabrikt generates controller interfaces and models from the OpenAPI spec
- `./gradlew jooqCodegen` — jOOQ generates table/record classes from the live PostgreSQL schema

Re-run these after changing the OpenAPI spec or DB schema respectively.

### OpenAPI Contracts

The OpenAPI source lives in `contracts/todo/` as a multi-file spec (paths, schemas, parameters split into individual files). `src/main/resources/openapi/todo.yaml` is the committed bundle output — do not edit it directly.

After changing the contract source files, run `./contracts/build.sh` to rebundle, then `./gradlew generateApi` to regenerate Kotlin code.
